## 一、索引模块

### **1. MySQL 为什么用 B+ 树做索引，而不用 B 树或红黑树？**

- **B+ 树非叶子节点只存键值**，不存数据，每页能存储更多键 → 树更矮 → 磁盘 IO 次数更少
- **叶子节点用双向链表连接** → 范围查询极快，只需找到起点后顺序遍历即可
- 红黑树是二叉树，深度大，磁盘 IO 次数多，不适合大规模数据存储
- 千万级数据 B+ 树只需 3~4 层，3 次 IO 即可定位

---

### **2. 聚簇索引和非聚簇索引（二级索引）的区别？**

| 对比维度 | 聚簇索引 | 非聚簇索引（二级索引） |
|---------|---------|---------------------|
| 叶子节点内容 | 存储**整行数据** | 存储**索引列 + 主键 ID** |
| 数量限制 | 每表只能有一个 | 每表可以有多个 |
| 默认创建 | 主键自动创建 | 手动创建 |
| 查询完整行 | 直接获取 | 需回表 |

---

### **3. 什么是回表？如何避免？**

**回表**：通过二级索引查到主键 ID 后，再回到聚簇索引中查找完整行数据的过程，多了一次 B+ 树遍历。

**避免方式**：使用**覆盖索引**——将查询所需的所有字段都建在联合索引中，让索引覆盖所有 SELECT 列和 WHERE 条件列。EXPLAIN 的 Extra 列出现 `Using index` 表示成功避免回表。

---

### **4. 联合索引的「最左前缀原则」是什么？**

联合索引 `(a, b, c)` 的查询必须从最左列开始**连续匹配**，否则索引失效：

- `WHERE a=1` → 走索引
- `WHERE a=1 AND b=2` → 走索引
- `WHERE a=1 AND b=2 AND c=3` → 走索引
- `WHERE b=2` → **不走索引**（跳过了 a）
- `WHERE a=1 AND c=3` → **只用 a，c 不走索引**（跳过了 b）

---

### **5. 索引失效的常见场景有哪些？（六大场景）**

| 场景 | 示例 | 原因 |
|------|------|------|
| 对索引列使用函数 | `WHERE YEAR(create_time)=2024` | 索引存的是原始值 |
| 左模糊查询 | `WHERE name LIKE '%张三'` | B+ 树无法定位前缀 |
| 隐式类型转换 | `WHERE phone=13800138000`（phone 是 varchar） | 触发 CAST 函数 |
| OR 中有非索引列 | `WHERE a=1 OR b=2`（b 无索引） | 需全表扫描 |
| 负向查询 | `!=`、`NOT IN`、`NOT EXISTS` | 无法利用索引定位 |
| 联合索引跳最左列 | `WHERE b=1`（联合索引 (a,b)） | 违反最左前缀 |

---

### **6. 什么是覆盖索引？它有什么好处？**

查询所需的**所有字段**（SELECT + WHERE + JOIN）都包含在同一个索引中，无须回表查询聚簇索引。

- 减少一次 B+ 树遍历，IO 减半
- 减少数据访问量，QPS 可翻倍
- Extra 列显示 `Using index` 即为覆盖索引

---

### **7. 为什么主键推荐自增 ID 而不是 UUID？**

- **自增 ID**：按顺序追加到 B+ 树末尾，无页分裂，写入效率高
- **UUID**：随机分散插入，频繁导致**页分裂**，产生索引碎片，写放大 2~3 倍
- 自增 ID 占用空间小（8 字节），UUID 占用大（36 字节或 16 字节），影响二级索引存储

## 二、事务与 MVCC 模块

### **8. 事务的 ACID 四大特性，底层分别如何实现？**

| 特性 | 含义 | 底层实现 |
|------|------|----------|
| **原子性 (Atomicity)** | 事务要么全成功，要么全回滚 | **undo log** 记录旧值用于回滚 |
| **一致性 (Consistency)** | 事务前后数据完整性约束不变 | A + I + D 共同保证 |
| **隔离性 (Isolation)** | 并发事务之间互不干扰 | **MVCC + 锁机制** |
| **持久性 (Durability)** | 事务提交后数据永久保存 | **redo log** 保证崩溃恢复 |

---

### **9. 数据库的四大隔离级别，分别解决了什么问题？**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|:---:|:-------:|:---:|
| READ UNCOMMITTED（读未提交） | ✗ | ✗ | ✗ |
| READ COMMITTED（读已提交） | ✓ | ✗ | ✗ |
| **REPEATABLE READ（可重复读，MySQL 默认）** | ✓ | ✓ | ✓（MVCC+间隙锁） |
| SERIALIZABLE（串行化） | ✓ | ✓ | ✓ |

> ✗ = 未解决，✓ = 已解决

---

### **10. 脏读、不可重复读、幻读的区别是什么？**

- **脏读**：读到其他事务**未提交**的"草稿"数据，对方可能回滚导致读到无效数据
- **不可重复读**：同一事务内两次读取**同一行**数据，结果不同（被其他事务 UPDATE 并提交）
- **幻读**：同一事务内两次**范围查询**，结果行数不同（被其他事务 INSERT 或 DELETE 并提交）

---

### **11. MVCC 是什么？它的实现原理？**

**MVCC（多版本并发控制）** 是一种让读写不互相阻塞的并发控制机制。

**实现原理**：
- 每行数据隐藏两列：`trx_id`（最近修改的事务 ID）、`roll_pointer`（回滚指针）
- 通过 `roll_pointer` 和 undo log 形成**版本链**
- 每次查询时生成 **Read View**，根据事务 ID 判断版本链中哪些数据对该事务可见
- 读操作读历史版本（快照读），写操作加锁（当前读），**读写互不阻塞**

---

### **12. InnoDB 和 MyISAM 的核心区别？**

| 维度 | InnoDB（默认引擎） | MyISAM |
|------|------------------|--------|
| 事务支持 | 支持 | 不支持 |
| 外键 | 支持 | 不支持 |
| 锁粒度 | 行级锁 | 表级锁 |
| 崩溃恢复 | 支持（redo/undo log） | 不支持 |
| 索引结构 | 聚簇索引 | 非聚簇索引 |
| 适用场景 | 高并发 OLTP（99% 线上场景） | 读多写少，已逐步淘汰 |

## 三、锁模块

### **13. 行锁、表锁、间隙锁、临键锁的区别？**

| 锁类型 | 锁定范围 | 说明 |
|--------|---------|------|
| **行锁** | 锁定单行记录 | 并发高，必须走索引否则退化为表锁 |
| **表锁** | 锁定整张表 | 并发低，MyISAM 默认使用 |
| **间隙锁** | 锁定索引记录之间的空隙 | 防止幻读，RR 隔离级别特有 |
| **临键锁 (Next-Key Lock)** | 行锁 + 该行之前的间隙锁 | InnoDB 防幻读的核心，RR 级别默认使用 |

---

### **14. 死锁是如何产生的？如何避免？**

**产生原因**：两个事务互相等待对方持有的锁，形成循环等待。
- 例如：事务 A 锁了行 1 等行 2，事务 B 锁了行 2 等行 1

**避免方法**：
- 固定加锁顺序，所有事务按相同顺序访问资源
- 保持事务短小精悍，减少锁持有时间
- 合理创建索引，防止行锁升级为表锁
- 设置 `innodb_lock_wait_timeout` 超时参数

## 四、日志模块

### **15. redo log、undo log、binlog 三者的区别？**

| 日志 | 所属层级 | 作用 | 记录内容 |
|------|---------|------|---------|
| **redo log** | InnoDB 引擎层 | 崩溃恢复，保证持久性 | 物理日志：记录数据页的修改 |
| **undo log** | InnoDB 引擎层 | 事务回滚 + MVCC | 逻辑日志：记录修改前的旧值 |
| **binlog** | Server 层 | 主从复制 + 数据恢复 | 逻辑日志：记录 SQL 语句或行变更 |

---

### **16. 一条 UPDATE 语句的完整执行流程（两阶段提交）？**

```
执行器调用引擎 → 在 Buffer Pool 中更新数据页
→ 写 undo log（记录旧值，用于回滚）
→ 写 redo log（prepare 状态）
→ 写 binlog
→ redo log 改为 commit 状态（两阶段提交完成）
→ 后台线程异步刷脏页到磁盘
```

**两阶段提交**确保 redo log 和 binlog 的一致性，保证崩溃恢复后主从数据一致。

## 五、SQL 优化模块

### **17. EXPLAIN 执行计划的核心字段含义？**

| 字段 | 含义 | 关注点 |
|------|------|--------|
| **type** | 访问类型 | `system > const > eq_ref > ref > range > index > ALL`<br>`ALL` = 全表扫描，需要优化 |
| **key** | 实际使用的索引 | NULL 表示未使用索引 |
| **rows** | 预估扫描行数 | 越小越好 |
| **Extra** | 额外信息 | `Using index`（覆盖索引，好）<br>`Using filesort`（文件排序，差）<br>`Using temporary`（临时表，差） |

---

### **18. DELETE、TRUNCATE、DROP 三者的区别？**

| 对比维度 | DELETE | TRUNCATE | DROP |
|---------|--------|----------|------|
| 操作粒度 | 可加 WHERE 删除指定行 | 删除表中全部数据 | 删除整个表 |
| 是否可回滚 | 可回滚（事务内） | 不可回滚 | 不可回滚 |
| 执行速度 | 慢（逐行删除并记日志） | 快（删除表后重建） | 快 |
| 表结构 | 保留 | 保留 | 一并删除 |
| 触发器 | 触发 | 不触发 | 不触发 |

## 六、基础概念模块

### **19. CHAR 和 VARCHAR 有什么区别？**

| 对比维度 | CHAR | VARCHAR |
|---------|------|---------|
| 存储方式 | 定长，不足用空格填充 | 变长，按实际长度存储 |
| 存储空间 | 可能浪费空间 | 更节省空间 |
| 读取速度 | 更快（无长度计算） | 略慢（需读取长度头） |
| 最大长度 | 255 字符 | 65535 字节 |
| 适用场景 | 固定长度字段（如 MD5、国家代码） | 长度不固定字段（如姓名、地址） |

---

### **20. JOIN 有哪些类型？分别返回什么数据？**

| JOIN 类型 | 返回结果 |
|-----------|---------|
| **INNER JOIN** | 两表中**都有匹配**的行 |
| **LEFT JOIN** | **左表全部行** + 右表匹配行，右表无匹配填 NULL |
| **RIGHT JOIN** | **右表全部行** + 左表匹配行，左表无匹配填 NULL |
| **CROSS JOIN** | 两表的**笛卡尔积**（每行和每行组合） |

> MySQL 原生不支持 `FULL JOIN`，可用 `LEFT JOIN UNION RIGHT JOIN` 模拟。
